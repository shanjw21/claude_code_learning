# Code Review 检查清单

## 审查流程

### 1. 理解意图
- 这个 PR/变更要解决什么问题？
- 如果意图不明确，先阅读相关文件了解上下文

### 2. 检查项

**正确性：**
- 逻辑是否有边界条件遗漏？
- 错误处理是否充分？（网络错误、文件不存在、超时）
- 是否有明显的安全问题？（SQL 注入、XSS、硬编码密钥）

**可维护性：**
- 函数/方法是否过长？（超过 50 行值得拆分）
- 变量命名是否清晰？
- 是否有重复代码可以提取？
- 是否引入了不必要的复杂度？

**测试：**
- 变更是否有对应的测试？
- 测试是否覆盖了边界情况？
- 测试名称是否描述了行为？

### 3. 输出格式

```
## 总体评价
[一段话总结这个变更的质量]

## 问题
- [严重/一般/建议] 具体问题描述

## 亮点
- 做得好的地方
```

## 注意事项

- 区分"必须改"和"建议改"
- 给出具体原因，不要只说"不好"
- 如果代码整体质量高，先肯定优点
